# AirPack Specification v1.1

## Dropdown Generation

AirLock generates the first two dropdowns from AirPack metadata.

### Dropdown 1 — Letter Type

Generated from:

```json
{
  "Title": "Legal Strict Facts",
  "MainType": "Appeal"
}
```

Displayed as:

```text
Legal Strict Facts > Appeal
```

Rules:

* Unique Title/MainType combinations produce unique entries.
* Multiple SubTypes may exist under the same Title/MainType.
* Title should represent the workflow family.
* MainType should represent the workflow category.

Examples:

```text
Legal Letters > Demand
Legal Letters > Appeal

Legal Strict Facts > Appeal

Insurance Workflows > Coverage Review

HOA Letters > Complaint
```

---

### Dropdown 2 — Letter Sub-Type

Generated from:

```json
{
  "Label": "Insurance Claim Appeal (Strict Facts)"
}
```

Displayed as:

```text
Insurance Claim Appeal (Strict Facts)
```

Rules:

* User-facing.
* Human readable.
* Does not need to be unique globally.
* Should clearly describe the workflow.

---

### Dropdown 3 — Escalation Level

Not controlled by AirPack.

Generated by runtime.

Current values:

```text
Initial Inquiry
Reminder
Demand
Final Notice
Intent to Escalate
```

Mapped through PromptMappings.

AirPacks should expect the selected escalation level to be injected through:

```text
{Tone}
```

---

### Dropdown 4 — Letter Length

Not controlled by AirPack.

Generated by runtime.

Current values:

```text
Brief
Short
Medium
Extended
Full
```

Mapped through PromptMappings.

Injected through:

```text
{Length}
```

# Formatting Templates (v1.1)

AirPacks may optionally define:

```json
{
  "HeaderTemplate": "...",
  "RecipientTemplate": "...",
  "FooterTemplate": "..."
}
```

Purpose:

Move document structure out of the model and into the workflow definition.

---

## HeaderTemplate

Example:

```json
"HeaderTemplate":
"{SenderName}
{SenderAddress}
{SenderCityStateZip}
{Date}"
```

Expected runtime behavior:

1. Runtime resolves placeholders.
2. Completed header is inserted into prompt.
3. Model should preserve header structure.

---

## RecipientTemplate

Example:

```json
"RecipientTemplate":
"{RecipientName}
{RecipientAddress}
{RecipientCityStateZip}"
```

---

## FooterTemplate

Example:

```json
"FooterTemplate":
"Sincerely,

{SenderName}"
```

---

# Placeholder Expansion

AirLock runtime should eventually support:

```text
{SenderName}
{SenderAddress}
{SenderCityStateZip}

{RecipientName}
{RecipientAddress}
{RecipientCityStateZip}

{Date}
```

before prompt submission.

Models should receive resolved values, not raw tokens.

---

# Fact Preservation Mode

AirPacks may define stricter drafting behavior.

Recommended workflow family:

```text
Legal Letters

Legal Strict Facts

Insurance Strict Facts

Compliance Strict Facts
```

Characteristics:

* Placeholder-first
* No inferred facts
* No inferred dates
* No inferred status updates
* Names treated as immutable data

Preferred behavior:

```text
Missing Fact
    ↓
Placeholder
```

Avoid:

```text
Missing Fact
    ↓
Assumption
```

---

# AirPack Design Principle

AirPack authors control:

* Workflow family (Title)
* Workflow category (MainType)
* Workflow selection (Label)
* Prompt strategy
* Formatting strategy
* Fact preservation strategy

AirLock runtime controls:

* Escalation levels
* Length options
* Model invocation
* Storage
* Import/export
* Database integration

**Runtime stays fixed. Workflow behavior lives in the AirPack.**
